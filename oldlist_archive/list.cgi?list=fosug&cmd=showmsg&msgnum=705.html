Run phpinfo() and see what the settings show. [0]

Of particular interest is that you look at the php.ini file location that your
php is looking at.  There are often more than one location that these show up
in and if you happen to be looking in the wrong location and changing settings
there, you'll develop a flat spot on your head.

If you find that register_globals and display_errors are both off, you've
worked on the wrong ini file.

If you find that you're using the right file on the CLI, and that through your
web server you have the same problem, make sure that there's no settings in
the .htaccess file in the directory you're interested in.

As to the NULL return on the ini_get calls, this is as designed.  If the
values aren't explicitly set in the php.ini file, then you'll get NULL string
returns.  The best way around this is to cast the values as integers when
reading, a la:

  print 'register_globals = ' . ((int) ini_get('register_globals')) . "\n";

which should yeild the desired 0|1 that you're looking for.

If you don't want to mess with the full-blown register_globals mojo, you can
simply create an include file (or find one that's used by every page) and
place a call to import_request_variables [1] or better yet, to extract [2].

<soapbox>
If you're running an application that still *requires* the use of
register_globals, you should consider another application VERY SERIOUSLY.  
</soapbox>
