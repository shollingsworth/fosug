Kinda, sorta, yes.  Theres a whole lot more detail that you need to put into
this message like what's available on the backend (e.g. PERL, PHP, Python,
...) and what *exactly* you wish to serve up - dynamic content, static web
pages, ... .

The downside on any apache auth module is that the credentials get cached by
the browser and then are sent with each request to the server as part of the
client headers.  If you're not using https for *all* requests to the server,
the credentials get sent over the wire base64 encoded (which you can decode
fairly quickly by hand).  The reason that this happens is that you're giving
up the authorization mechanism to the server and not keeping it at your
application level.

> Does anyone have a better solution to suggest?

Why, yes.  I would suggest using a scripting language or some sort of portal
that can let you control per user access via sessions.  This allows you to
terminate the session based on a number of criterion (user logout, expiry
time, insecure transmission) and only requires the transmission of the session
key (hopefully over an https connection).

I'll spare the list, but I have an ldap auth module for php that you can have
(it's for use against NDS, but should work for any LDAPv2 compliant server -
which I believe AD is) and modify to suit your needs if you would like.  I
pretty much abandoned Perl for scripting at work to standardize on PHP for my
colleague's sanity.  You can do the same things with Perl, and I think I have
the same code for Perl somewhere.
