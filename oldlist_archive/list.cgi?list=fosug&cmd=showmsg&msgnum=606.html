> Besdies if I was running productivity software, I wouldn't necessarily
> want to use a google, or sun server, but install the app on a local
> web server so my information isn't exposed to the public net. Or at
> least much less exposure anyway.

I think this is better done with any number of corporate solutions, a la
Novell Zen for Desktops.  The cost is not bad when compared to manual
installation on several machines (I don't actually know what the "break even"
point would be, but imagine it can't be much more than about 10) and the
feature set extends across your enterprise and not just to a single
application (or suite of applications).

Besides that, how does google/sun make *any* money by allowing you to run the
server hosting/serving the data and applications?  

The last time I looked at the ASP business model, they leased you applications
that you used and then had to lease again for the next "time period" based on
your usage requirement.  The thought being that you always get the newest
release of the software and only pay as you need the software (M$ was betting
that nobody could live without Orifice).  The two problems I see with this
format are: 1) the assumption of high speed bandwidth for the "entire"
installed base of the M$ Orifice suite, and; 2) the belief that people are
willing to wade through the upgrade process *more* frequently than they do now
for incremental increases in functionality.

For point one, there is definitely a better business case today than five
years ago, but the SOHO space is still fluctuating between hi-speed and dialup
internet.  As for the home user, dial-up is still the big winner just from the
infrastructure standpoint.

On two, if you're not using broadband, are you going to wait for the download
each time there's a bug fix/enhancement to the code?  What if you're paying
line charges as a business?  Do you want to upgrade everytime the feature set
changes?
