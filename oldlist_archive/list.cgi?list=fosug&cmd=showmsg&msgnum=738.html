Are you talking about Postfix or qmail on the user id part?

Postfix requires two groups and one user.  Qmail uses seven users and two
groups.  The user count is higher in qmail but so what?  None of them are
interactive logins and it simply provides greater program separation between
the components of qmail.  This adds security as no program implicity trusts
any other program in the qmail system and only one program runs with escalated
privileges.

As for mbox, qmail supports mbox just fine.  It's silly to use qmail with mbox
unless you have client requirements for mbox files.  There are many reasons to
*not* use mbox files, but if you must, qmail can do it just fine (by default
even).

> My understanding is that qmail's default delivery is actually NOT Maildir,

That is correct.  If you choose not to give qmail direction, you'll get mbox

format his mail exists as instead of making the admin choose for all users.

I will never use mbox again if I don't have to because I've been the victim of

being that I can use find to sort my mail by date automatically.

One-to-many delivery in qmail works as you say it should, for recipients in
the CC field of the mail.  However, for emails that use BCC for recipients,
one-to-many delivery can be done either as a single message with many RCPT TO
commands or as multiple messages:

     [0] Addresses that do not appear in the message headers may appear in the
     RCPT commands to an SMTP server for a number of reasons.  The two most
     common involve the use of a mailing address as a "list exploder" (a
     single address that resolves into multiple addresses) and the appearance
     of "blind copies".  Especially when more than one RCPT command is
     present, and in order to avoid defeating some of the purpose of these
     mechanisms, SMTP clients and servers SHOULD NOT copy the full set of RCPT
     command arguments into the headers, either as part of trace headers or as
     informational or private-extension headers.  Since this rule is often
     violated in practice, and cannot be enforced, sending SMTP systems that
     are aware of "bcc" use MAY find it helpful to send each blind copy as a
     separate message transaction containing only a single RCPT command.

So as I read the RFC, it says that if you're sending bcc, you may need to send
a single message per recipient to prevent mailers that, 'violate [bcc] in
practice]' from giving out the recipient list.  It sounds like qmail is right
on spot there.

To hopefully put an end to this note, I run qmail and I have run postfix in
the very recent past.  I chose qmail over postfix at this time and place
because I understand it better than postfix and enjoy the flexibility that
qmail allows my admin staff at work.  Postfix's ties to sendmail compatibility

for spam through an external program although the UCE controls are quite cool.
